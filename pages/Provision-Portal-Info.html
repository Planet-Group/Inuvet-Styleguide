Provision-Portal — Dokumentation

Dieses Dokument beschreibt das Provision-Portal für Tierärzte im inuvet-Partnerprogramm. Es richtet sich an Entwickler, die das Portal implementieren oder erweitern möchten. Im Mittelpunkt steht die Frage: Wie löst ein Tierarzt angesammelte Provisionen ein — als Barauszahlung oder als Sachprämie?

Grundidee

Tierärzte, die Produkte von inuvet empfehlen, erhalten pro vermittelter Bestellung eine Provision. Diese Provisionen sammeln sich im Portal an und können auf zwei Wegen eingelöst werden:

  1. Sofortauszahlung — Die verfügbare Provision wird direkt auf das hinterlegte Bankkonto überwiesen.
  2. Prämie wählen — Die Provision wird gegen eine physische Prämie oder einen Gutschein eingetauscht, sofern der Wert ausreicht.

Das Portal ist keine Shop-Seite — der Tierarzt kauft nichts, er löst ein. Der Provisionsbetrag ist die „Währung", Prämien haben einen definierten Provisionswert.

Die vier Views

Das Portal ist als Single-Page Application (SPA) umgesetzt. Die Seite lädt einmalig, alle Inhalte werden clientseitig über eine zentrale render()-Funktion gerendert. Es gibt vier Views:

ViewZustand (currentPage)Beschreibung
Portal 'portal' Übersicht: Hero mit Provisionsbetrag + Ring, Prämien-Kachelraster
PDP 'pdp' Detailansicht einer Prämie — Galerie, Beschreibung, Akkordeon, Warenkorb-Aktion
Checkout 'checkout' Lieferadresse, Versandart, Bestellübersicht — Einlösung bestätigen
Erfolg 'success' Bestätigungsseite nach Einlösung (Prämie) oder Auszahlung (Sofortauszahlung)

Navigation zwischen Views erfolgt ausschließlich über die JS-Funktionen openPDP(id), backToPortal(), goToCheckout() und completeCheckout(). Jede Navigation scrollt automatisch zum Seitenanfang (window.scrollTo(0, 0)).

Hero-Bereich

Die Portal-Ansicht öffnet mit einem dreiteiligen Hero:

  • Links: Beschriftung „Deine aktuelle Provision" (auf Desktop, auf Mobilgeräten als eigener Block)
  • Mitte: Animierter Provisionsbetrag (countUp()) + Button „Sofort auszahlen"
  • Rechts: SVG-Fortschrittsring mit Prozentzahl — zeigt, wie weit die Provision zur nächstgünstigsten noch nicht einlösbaren Prämie reicht

Darunter erscheint eine dreispaltige Metazeile: Basis der Berechnung links, „oder" in der Mitte, rechts ein dynamischer Hinweis — entweder „Noch X,XX € bis [Prämienname]" als Link zur PDP oder „Alle Prämien einlösbar".

Fortschrittsring im Hero

Der Hero-Ring verwendet einen SVG-Kreis mit stroke-dasharray / stroke-dashoffset. Der Ziel-Offset wird serverseitig in das data-offset-Attribut geschrieben und via animateHeroRing() nach dem Render per CSS-Transition animiert:

// Transition auf .--in-view (nicht auf [data-animate]) —
// damit opacity:0 sofort greift und die Animation nur einmal vorwärts läuft
[data-animate].--in-view {
  opacity: 1; transform: none;
  transition: opacity 1s …, transform 1s …;
}

Prämien-Modell

Jede Prämie ist ein Objekt im Array PREMIUMS mit diesen Pflichtfeldern:

{
  id:       Number,    // eindeutige ID
  name:     String,    // Anzeigename (Kachel + PDP + Warenkorb)
  price:    Number,    // Provisionswert in Euro
  desc:     String,    // Kurzbeschreibung (Tile)
  longDesc: String,    // Ausführliche Beschreibung (PDP-Akkordeon)
  image:    String,    // optional — Pfad zum Produktbild
  voucher:  Object,    // optional — { brand: String, value: String }
}

Das Feld voucher ist optional und exklusiv zu image: Hat eine Prämie ein voucher-Objekt, wird statt eines Fotos eine Gutscheinkarte (.voucher-card) gerendert — sowohl auf der Kachel als auch auf der PDP. Gutscheinkarten sind eine bewusste Ausnahme zur globalen border-radius: 0-Regel (Wert: 1.25rem).

Die Prämien werden von teuer nach günstig sortiert. Die Sortierung liegt im PREMIUMS-Array selbst — keine dynamische Sortierung zur Laufzeit. Für den Hinweis im Hero wird das Array jedoch aufsteigend (günstig → teuer) durchsucht, um die nächste noch nicht einlösbare Prämie zu finden (nextNonAffordable()).

Einlösbarkeit — Logik

Eine Prämie gilt als einlösbar, wenn die verfügbare Provision (= provisionTotal minus Gesamtbetrag der Prämien im Warenkorb) mindestens dem Preiswert der Prämie entspricht.

provisionAvailable() = provisionTotal - cartTotal()
canAfford(p)         = provisionAvailable() >= p.price

Eine Prämie im Warenkorb „reserviert" die Provision bereits — andere teurere Prämien können dadurch unerschwinglich werden. Der Ring und alle Badges aktualisieren sich bei jeder Warenkorb-Änderung neu (render() wird immer vollständig aufgerufen).

Drei Zustände pro Kachel und PDP:

ZustandBadgeWarenkorb-Button
Einlösbar .badge --sale „Einlösbar" „In den Warenkorb" (.btn --primary)
Nicht genug Provision .badge --dark „Noch X,XX €" Deaktiviert „Noch X,XX € fehlend"
Im Warenkorb — (kein Badge) „Entfernen" / „Aus Warenkorb entfernen" (.btn --secondary)

Warenkorb

Der Warenkorb (cart) ist ein einfaches Array von Prämien-IDs. Da Provisionen eine endliche Ressource sind, wird beim Hinzufügen einer Prämie geprüft, ob sie erschwinglich ist — es gibt keine Mengenauswahl. Jede Prämie kann nur einmal eingelöst werden (kein Duplikat-Check nötig, da der Button nach dem Hinzufügen auf „Entfernen" wechselt).

Der Cart-Drawer rendert sich bei jedem render()-Aufruf vollständig neu. Bei leerem Warenkorb erscheint ein .empty-state. Der Drawer zeigt die verbleibende Provision nach Einlösung transparent unter dem Gesamt-Betrag.

Sofortauszahlung

Der Button „Sofort auszahlen" öffnet ein Modal (#auszahlungModal) mit IBAN und Umsatzsteuer-ID. Nach Bestätigung:

  1. provisionTotal wird auf 0 gesetzt
  2. cart wird geleert
  3. lastOrder erhält das Flag payout: true und den ausgezahlten Betrag
  4. View wechselt zu 'success'

Die Erfolgsseite unterscheidet über lastOrder.payout, ob sie eine Prämien-Einlösung oder eine Auszahlung bestätigt — der Inhalt ist jeweils verschieden.

Animationen

Das Portal nutzt drei separate Animationssysteme, die alle nach dem Render in requestAnimationFrame gestartet werden:

FunktionZielBeschreibung
countUp(target) #provisionAmount Zählt den Euro-Betrag von 0 auf den Zielwert hoch (cubic-ease-out, 1,8 s)
animateHeroRing() .provision-hero__ring-fill Setzt strokeDashoffset auf den Zielwert — CSS-Transition übernimmt die Animation (1,6 s)
initRings() .tile__ring-fill je Kachel IntersectionObserver (threshold 0,4) — animiert Ring sobald Kachel sichtbar ist; einmalig pro Element
initScrollAnimations() [data-animate] Globaler Observer aus inuvet.js — fügt .--in-view hinzu, wodurch opacity + translateY animiert werden

Wichtig: Die CSS-Transition für [data-animate] liegt ausschließlich auf dem .--in-view-Zustand — nicht auf dem Ausgangszustand. Würde die Transition auf [data-animate] stehen, wäre die opacity-Rückkehr von 1 auf 0 beim Render sichtbar, bevor der Observer greift.

Mockup-FAB

Der Floating-Action-Button (⚙ Mockup-Kontext) ermöglicht zwei Simulationen ohne Code-Eingriff:

  • Provision simulieren — setzt provisionTotal auf einen beliebigen Wert und rendert neu
  • Gutschein-Karte anpassen — ändert voucher.brand und voucher.value der Amazon-Prämie live (nützlich für Demo-Zwecke mit anderen Marken)

Der FAB-Panel nutzt ausschließlich mockup-ui.css-Klassen (.mockup-fab-panel, .mockup-btn, .mockup-fab-panel__field) — keine inuvet.css-Klassen, gemäß Goldener Regel 12.

Technische Implementierung — Übersicht

Datenmodell (Backend)

  1. Provisions-Konto: Pro Tierarzt-Account wird der aktuelle Provisionssaldo gespeichert. Jede erfolgreiche Empfehlung erhöht ihn.
  2. Prämien-Katalog: Prämien werden serverseitig verwaltet (Preis, Verfügbarkeit, Bild-URL, ggf. Gutschein-Daten). Das Frontend erhält die Liste beim Laden der Seite.
  3. Einlösung: Eine Prämien-Einlösung erstellt einen Einlösungsauftrag (Lieferadresse, Prämien-IDs, Zeitstempel) und bucht den Provisionswert vom Konto ab.
  4. Auszahlung: Eine Auszahlungsanfrage erstellt einen Buchungssatz (IBAN, Betrag, Uststatus) und setzt das Konto auf 0. Auszahlung erfolgt manuell oder via SEPA-Batch.

Frontend-Integration

  1. Seitenload: Portal fragt Provisions-Saldo und Prämien-Katalog ab — beides wird in provisionTotal und PREMIUMS geschrieben, dann render().
  2. Warenkorb-Persistenz: Im Prototyp ist der Warenkorb flüchtig. In Produktion sollte er zumindest session-basiert zwischengespeichert werden (kein Verlust bei Tab-Schließen).
  3. Checkout-Submit: Lieferadresse + Prämien-IDs werden an eine POST /einloesungen-Route gesendet. Das Backend prüft erneut, ob die Provision noch ausreicht (Race Condition: mehrere Tabs, Admin-Korrekturen).
  4. Auszahlung-Submit: IBAN + Betrag an eine POST /auszahlungen-Route. Backend validiert IBAN und erstellt Buchungsauftrag.
  5. E-Mail-Bestätigung: Nach Einlösung oder Auszahlung erhält der Tierarzt eine Bestätigungs-E-Mail an die hinterlegte Praxis-Adresse.

Responsive Verhalten

Unterhalb von 700px Viewport-Breite blendet der Hero-Ring und die linke Label-Spalte aus. Der Provisionsbetrag rückt in die Bildschirmmitte, der Titel erscheint als .provision-hero__mobile-title darüber. Die Metazeile (.provision-hero__sub) ist auf Mobilgeräten ausgeblendet.
