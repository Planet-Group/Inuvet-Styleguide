Persönliches Angebot: Strategie, UX-Konzept und technische Spezifikation

Das Persönliche Angebot ist ein zentrales Retention- und AOV-Instrument im inuvet Shop. Dieses Dokument beschreibt die strategische Rationale, die UX-Entscheidungen und die technischen Anforderungen für Entwicklung und Integration — als gemeinsame Grundlage für Produkt, Marketing und IT.

Strategische Zielsetzung

Wiederkäufer sind für inuvet die wertvollste Kundensegment: ihre CAC-Amortisation ist abgeschlossen, ihre Kauffrequenz vorherseh­bar, ihr LTV deutlich höher als der eines Erstkäufers. Das Persönliche Angebot zielt auf zwei messbare Hebel:

  • Erhöhung des Average Order Value (AOV): Das Naturalrabatt-System schafft einen mengen­abhängigen Stufenreiz. Kunden, die ohnehin nachbestellen, erhöhen ihre Bestellmenge, um die nächste Gratis-Schwelle zu erreichen — ohne dass ein Gutscheincode kommuniziert werden muss.
  • Verkürzung des Wiederkauf-Zyklus: Ein vorausgefülltes, personalisiertes Angebot senkt die kognitive Hürde der Nachbestellung drastisch. Der Kunde muss nicht mehr suchen, vergleichen oder seinen letzten Einkauf rekonstruieren.

Das Naturalrabatt-System als Marketing-Mechanik

Klassische Mengenrabatte werden als Preisreduzierung wahrgenommen — was die Preiserwartung langfristig senkt und Margen untergräbt. Der Naturalrabatt funktioniert psychologisch anders: Der Kunde zahlt den vollen Preis für seine Einheiten und erhält zusätzlich Produkte gratis. Das Framing ist „mehr bekommen" statt „weniger zahlen" — ein entscheidender Unterschied für die Markenwahrnehmung.

„Naturalrabatt positioniert die Großpackung nicht als Sparangebot, sondern als Premium-Entscheidung. Der Kunde fühlt sich belohnt — nicht abgezockt." — inuvet Produkt-Team

UX-Konzept und Konversions-Architektur

Die Seite ist als geführter Entscheidungsprozess konzipiert, nicht als generischer Warenkorb. Jede UX-Entscheidung folgt einer klaren Conversion-Logik:

  • Vorausgefüllte Produktliste: Basierend auf den Bestelldaten der letzten sechs Monate werden Produkte und Mengen automatisch vorgeschlagen. Der Kaufaufwand reduziert sich auf eine einzige Bestätigung. Kauffrequenz und Bestellhistorie sind direkt am Produkt sichtbar — das erzeugt Vertrauen und macht die Empfehlung nachvollziehbar.
  • Tier-Upsell-Hint: An jedem Produkt wird inline angezeigt, wie viele Einheiten bis zur nächsten Gratis-Schwelle fehlen. Das schafft einen konkreten, unmittelbaren Handlungsimpuls ohne generisches Upsell-Banner.
  • Sticky Summary (Mobile): Auf Mobilgeräten bleibt die Zusammenfassungsbox am unteren Bildschirmrand fixiert, sobald der Bundle-Bereich im Viewport sichtbar ist. Die Gesamtersparnis ist jederzeit präsent — das senkt Abbruchraten bei längeren Angebotslisten.
  • Empfehlungsleiste: Produkte, die der Kunde früher bestellt hat, aber im aktuellen Angebot nicht vorhanden sind, werden als horizontale Scroll-Leiste unterhalb der Liste angeboten. Das sind keine generischen Cross-Sells, sondern datenbasierte Reaktivierungsimpulse.

Technische Architektur

Die gesamte Kalkulationslogik sollte serverseitig implementiert sein. Das Frontend rendert ausschließlich und bezieht alle Werte aus einer API-Antwort — keine Geschäftslogik im Client. Folgende Formeln beschreiben die serverseitige Berechnung:

Datenstruktur: Rabattstufen

Rabattstufen sind produktspezifisch konfigurierbar. Produkte ohne eigene Konfiguration erben den systemweiten Standard (defaultTiers). Die Auflösung erfolgt per Fallback-Logik serverseitig — kein Hardcoding im Client.

// Systemweiter Standard
defaultTiers = [
  { minQuantity: 24, freeItems: 5 },
  { minQuantity: 16, freeItems: 3 },
  { minQuantity:  8, freeItems: 1 },
  { minQuantity:  0, freeItems: 0 }  // Fallback-Stufe (immer erforderlich)
]

// Produkt mit eigener Staffelung (überschreibt Standard)
product.tiers = [
  { minQuantity: 10, freeItems: 2 },
  { minQuantity:  5, freeItems: 1 },
  { minQuantity:  0, freeItems: 0 }
]

// Produkt ohne eigene Staffelung → erbt Standard
product.tiers = null

Fallback-Logik

Die Auflösung ist einzeilig: produktspezifische Stufen haben Vorrang, fehlen sie, gilt der Standard. Diese Logik muss serverseitig bei jeder Angebotsberechnung angewendet werden.

getProductTiers(product) =
  product.tiers ?? defaultTiers

Gratisprodukte pro Produkt

Immer mit den aufgelösten Stufen des jeweiligen Produkts rechnen. Erste Stufe von oben, bei der qty ≥ minQuantity.

freeItems(product, qty) =
  getProductTiers(product).find(t => qty >= t.minQuantity).freeItems

Gesamtersparnis

Für jedes Produkt im Angebot: Gratisanzahl × Einzelpreis, berechnet mit den jeweiligen produktspezifischen Stufen. Über alle Produkte summieren.

totalSavings =
  Σ freeItems(p, p.qty) × p.price
  for each product p in bundle

Nächste Stufe & Upsell-Hinweistext

Auch hier immer die produktspezifisch aufgelösten Stufen verwenden — zwei Produkte können unterschiedliche Schwellenwerte und damit unterschiedliche Hinweistexte haben.

tiers    = getProductTiers(product)
nextTier = tiers.find(t => t.minQuantity > qty)
missing  = nextTier.minQuantity - qty
gained   = nextTier.freeItems - freeItems(product, qty)

// → „{missing} Stück hinzufügen für weitere {gained} Gratisprodukte"
// Kein Hinweis wenn nextTier === undefined (Maximalstufe erreicht)

Bestellübergabe und Buchhaltung

Gratisprodukte werden nicht als separate Warenkorbpositionen übergeben. Die Gesamtmenge (chargedQty + freeItems) wird zu dem Preis gebucht, der nur für die bezahlten Einheiten gilt. Der Naturalrabatt-Betrag wird als eigene Buchungsposition für die Buchhaltung geführt.

chargedQty = qty               // Berechnungsgrundlage: nur bezahlte Menge
shippedQty = qty + freeItems   // Tatsächlich versendete Menge
discount   = freeItems × price // Naturalrabatt-Betrag für Buchhaltung

Datenbasis und Personalisierung

Das Angebot basiert auf den Bestelldaten der letzten sechs Monate. Für die Implementierung sind folgende Datenpunkte pro Kunde erforderlich:

  • order_history — Produkt-IDs und Mengen der letzten 180 Tage
  • purchase_frequency — berechnete Kauffrequenz pro Produkt (Bestellungen / 6 Monate)
  • suggested_quantity — aufgerundete Menge auf nächste Tier-Schwelle, optional konfigurierbar

Produkte ohne Bestellhistorie erscheinen nicht im Hauptangebot, sondern ausschließlich in der Empfehlungsleiste. Die Empfehlungen können zusätzlich durch manuelle Kuratierung oder Collaborative Filtering ergänzt werden — das System ist dafür offen, ohne dass die Grundfunktion davon abhängt.

Konfiguration und Rollout

Für den initialen Launch empfiehlt sich ein segmentierter Rollout: zunächst Kunden mit mindestens drei Bestellungen in den vergangenen sechs Monaten (hohe Datenqualität, geringeres Risiko leerer Angebote). Die Tier-Schwellenwerte sollten initial konservativ gesetzt werden — lieber eine breite erste Stufe mit attraktivem Einstieg als zu hohe Hürden, die den Upsell-Effekt verhindern. KPIs für die erste Phase: AOV-Entwicklung, Anteil der Kunden die eine Tier-Stufe erreichen, und Wiederkauf-Rate innerhalb von 90 Tagen.
