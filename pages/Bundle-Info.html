Persönliches Angebot — Dokumentation

Dieses Dokument beschreibt die Funktionsweise des Persönlichen Angebots im inuvet Shopify-Shop. Zielgruppe ist die IT, die das System implementiert. Es geht ausschließlich um Logik, Datenflüsse und Berechnungs-Formeln — keine Marketing-Aspekte.

Überblick

Das Persönliche Angebot zeigt eingeloggten Kunden eine vorausgefüllte Produktliste, basierend auf ihrer Bestellhistorie der letzten 549 Tage (18 Monate). Pro Einzelprodukt-Position wird ein Naturalrabatt berechnet: zusätzliche Stück des jeweiligen Produkts werden bei Erreichen bestimmter Schwellen kostenlos zur Bestellung dazugegeben.

Für die Berechnung des Naturalrabatts gibt es zwei Modelle, von denen pro Produkt im Backend eines konfiguriert wird:

  • Kondition A (Default) — Naturalrabatt-Guthaben in € basierend auf dem Bestellwert der Produktzeile
  • Kondition B — Feste Anzahl Gratis-Stück basierend auf der bestellten Menge

Beide Tabellen sind global fix. Es gibt kein produktspezifisches Override der Werte — die Konfiguration pro Produkt beschränkt sich auf die Wahl A oder B.

Produkt-Modell: Einzelprodukt vs. Produktfamilie

Für die Naturalrabatt-Berechnung ist der Begriff Einzelprodukt zentral — er ist nicht dasselbe wie „Produktfamilie".

  • Produktfamilie — alle Darreichungsformen einer Indikation (z. B. Familie „Hypolene" = Pulver + Tabletten). Dient der Navigation, Kachel-Übersicht und Backend-Konfiguration (Kondition A/B).
  • Einzelprodukt — eine konkrete Darreichungsform in einer bestimmten Größe (z. B. „Hypolene Pulver 60 g", „Calmin balance Tabletten 60 Stück"). Entspricht einer Warenkorb- bzw. Order-Position in Shopify.

Naturalrabatt wird pro Einzelprodukt-Position berechnet — nie über eine Produktfamilie aggregiert. Hypolene Pulver und Hypolene Tabletten sind zwei getrennte Berechnungen; Mengen und Bestellwerte werden nicht über Darreichungsformen hinweg summiert.

Berechnungsregeln (gelten für beide Modelle)

  1. Die Berechnung erfolgt pro Einzelprodukt-Position (konkrete Darreichungsform + Größe) — jede Zeile im Warenkorb/Angebot wird unabhängig betrachtet.
  2. Als Gratis-Produkt wird immer das jeweilige Einzelprodukt selbst zusätzlich versendet (kein Cross-Selling, kein anderes Produkt).
  3. Anzahl Gratis-Stück ist ganzzahlig abgerundet (Math.floor). Bei Kondition A entstehende Restguthaben verfallen.
  4. Die gesamte Berechnungslogik gehört serverseitig in das Shopify-Backend (App / Function). Das Frontend rendert nur das Ergebnis.

Kondition A — Bestellwert in Guthaben

Kondition A ordnet einem Bestellwert (in €) ein Naturalrabatt-Guthaben (in €) zu. Die Anzahl Gratis-Stück berechnet sich als floor(Guthaben ÷ Einzelpreis).

Der Bestellwert ist Menge × Einzelpreis der jeweiligen Einzelprodukt-Position — nicht der Familie, nicht über mehrere Darreichungsformen summiert. Beispiel: 16× Hypolene Pulver und 8× Hypolene Tabletten ergeben zwei getrennte Bestellwerte; es gibt keinen gemeinsamen „Hypolene"-Bestellwert.

Die Tabelle ist eine Stufenlookup: Es gilt das Guthaben der höchsten erreichten Schwelle. Zwei Stufen können denselben Wert haben (Plateau-Bereiche).

Berechnung Kondition A

function lookupCreditA(orderValue) {
  let credit = 0;
  for (const tier of TABLE_A) {
    if (orderValue >= tier.threshold) credit = tier.credit;
    else break;
  }
  return credit;
}

function freeItemsA(quantity, unitPrice) {
  const credit = lookupCreditA(quantity * unitPrice);
  return Math.floor(credit / unitPrice);
}

Beispiel Kondition A

16× Hypolene Pulver à 39,90 € → Bestellwert nur für diese Position 638,40 € → erreichte Schwelle 500 € → Guthaben 60 € → floor(60 / 39,90) = 1 → 1 Stück Hypolene Pulver gratis.

Gegenbeispiel: Zusätzlich 8× Hypolene Tabletten à 35,90 € → separater Bestellwert 287,20 € für die Tabletten-Position — kein gemeinsamer Bestellwert mit dem Pulver.

Kondition B — Stückzahl zu Gratis-Anzahl

Kondition B ordnet einer Bestellmenge (Stück) eine feste Anzahl Gratis-Stück desselben Produkts zu. Es gibt zwei Spalten, abhängig vom Kunden-Status:

  • Standard: gilt für alle Kunden ohne speziellen Status
  • Gruppen: gilt für Kunden mit definiertem Gruppen-Status (siehe unten)

Berechnung Kondition B

function freeItemsB(quantity, customerType) {
  // TABLE_B ist aufsteigend nach minQty sortiert
  const tier = [...TABLE_B].reverse().find(t => quantity >= t.minQty);
  if (!tier) return 0;
  return customerType === 'group' ? tier.group : tier.standard;
}

Beispiele Kondition B

  • 30× Calmin balance Tabletten (Standard-Kunde) → Schwelle 30 → 6 Stück Calmin balance Tabletten gratis
  • 30× Calmin balance Tabletten (Gruppen-Kunde) → Schwelle 30 → 3 Stück gratis
  • 27× Calmin balance Tabletten (Standard-Kunde) → Schwelle 25 → 5 Stück gratis
  • 10× Calmin balance Tabletten (Gruppen-Kunde) → unter Schwelle 18 für Gruppen → 0 Stück gratis

Calmin balance Pulver und Calmin balance Tabletten sind getrennte Einzelprodukt-Positionen — die Stückzahl wird nicht über Darreichungsformen summiert.

Identifikation des Kunden-Status (für Kondition B)

Der „Gruppen"-Status wird in Shopify per Customer Tag markiert. Vorschlag für die Konvention:

  • group-customer — Kunde gehört zur Gruppe → es gilt Spalte „Gruppen"
  • ohne Tag → es gilt Spalte „Standard"

Der Tag wird beim Berechnungszeitpunkt abgefragt. Tag-Änderungen wirken bei der nächsten Berechnung — kein Caching pro Kunde.

Auswahl des Modells (Kondition A / B)

Im Backend wird pro Shopify-Produkt (bzw. pro Indikation in der Familien-Logik) festgelegt, ob Kondition A oder B gilt. Vorschlag für die Konfiguration: Product Metafield:

  • Namespace: inuvet
  • Key: pricing_model
  • Typ: single_line_text mit Validation (akzeptierte Werte: A, B)
  • Default (wenn Metafield nicht gesetzt): A
// Auflösung
const model = product.metafields.inuvet.pricing_model || 'A';

// Berechnung immer pro Order-Line-Item (Einzelprodukt-Position):
// orderValue = line.quantity * line.unit_price

Bundle-UI (Mockup / Frontend)

Seitenspezifische Logik liegt in pages/bundle.js (nicht inline). Globale Naturalrabatt-Funktionen in inuvet.js.

  • Initiales Angebot: maximal 4 Einzelprodukte aus dem aktiven Historien-Pool (sortiert nach Kauffrequenz).
  • Leerer Zustand: Wenn der Kunde alle vorgeschlagenen Positionen entfernt → Empty State mit „Wiederherstellen" (setzt das initiale Angebot zurück). Keine Empfehlungs-Schiene zum Nachfüllen.
  • Mobile: Summary-Card als kompakte Fixed-Bar am unteren Rand, sichtbar solange die Bundle-Section im Viewport ist (Titel, Sparzeile, CTA — ohne Produktlisting).
  • Sichtbarkeit der Section: einmalig beim Page-Load entschieden; bleibt sichtbar auch wenn der Kunde danach alle Rabatt-Positionen entfernt.

Live-Vorschau im Frontend („Upsell-Hint")

An jeder Produktzeile wird ein Hinweis angezeigt, wie viele Stück bis zur nächsten Schwelle fehlen, die tatsächlich mehr Gratis-Stück bringt. Plateau-Schwellen (gleicher Wert in Folgestufe) werden übersprungen — sonst wäre die Hint sinnlos („3 Stück hinzufügen für 0 weitere Gratisprodukte").

function getHint(product, quantity, customerType) {
  const model = product.pricing_model || 'A';
  const price = product.price;
  const currentFree = (model === 'A')
    ? freeItemsA(quantity, price)
    : freeItemsB(quantity, customerType);

  if (model === 'A') {
    for (const tier of TABLE_A) {
      if (tier.threshold <= quantity * price) continue;
      const neededQty = Math.ceil(tier.threshold / price);
      const newFree   = Math.floor(tier.credit / price);
      if (newFree > currentFree) {
        return { moreQty: neededQty - quantity, moreFree: newFree - currentFree };
      }
    }
    return null;
  }

  // Modell B
  for (const tier of TABLE_B) {
    if (tier.minQty <= quantity) continue;
    const newFree = customerType === 'group' ? tier.group : tier.standard;
    if (newFree > currentFree) {
      return { moreQty: tier.minQty - quantity, moreFree: newFree - currentFree };
    }
  }
  return null;
}

// → „{moreQty} Stück hinzufügen für {moreFree} weiteres Gratisprodukt"
// Plural-Form: „… für {moreFree} weitere Gratisprodukte" wenn moreFree > 1
// Kein Hinweis wenn Funktion null zurückgibt (Maximalstufe erreicht)

Anzeige-Bedingung: kein Angebot ohne Naturalrabatt

Die Bundle-Section wird nur angezeigt, wenn mindestens ein Produkt im vorgeschlagenen Angebot tatsächlich Gratis-Stück erzeugt. Andernfalls ergibt das Angebot keinen Mehrwert — der Kunde würde nur seine normale Bestellliste sehen, ohne Vorteil.

Zeitraum-Prüfung (18 Monate)

Die Prüfung läuft serverseitig vor dem Rendering auf Basis der Bestellhistorie der letzten 549 Tage (past_549_days / past18Months im Mockup):

  1. Erzeugt mindestens ein Einzelprodukt im Pool Gratis-Stück? → Bundle-Section anzeigen, Zeitraum-Label „letzte 18 Monate".
  2. Kein Produkt liefert Gratis-Stück → Bundle-Section nicht ausgeben (ggf. Redirect auf Standard-Account-Page).

Einmalige Entscheidung — kein Nachjustieren zur Laufzeit

Die Sichtbarkeit wird nur beim initialen Page-Load bestimmt. Entfernt der Kunde danach Produkte aus dem Bundle, bis kein Rabatt mehr übrig bleibt, bleibt die Section trotzdem sichtbar — ein plötzliches Verschwinden wäre für den Kunden verwirrend. Die Prüfung wird nicht wiederholt.

// Serverseitig vor dem Rendering (einmalig):
function resolveBundle(customer) {
  const bundle = buildBundle(customer.history_549);
  if (bundle.some(p => calcFree(p) > 0)) {
    return { show: true, bundle, period: '549' };
  }
  return { show: false };
}

const result = resolveBundle(customer);
if (!result.show) {
  // Bundle-Section nicht ausgeben
  // → ggf. Redirect auf Standard-Account-Page
}

Vorausgefüllte Produktliste

Das initiale Angebot enthält die vier häufigst gekauften Einzelprodukte aus dem 18-Monate-Pool, sortiert nach Kauffrequenz (häufigster Kauf oben). Die Vorschlagsmenge pro Position ergibt sich aus der Kauffrequenz, sinnvoll aufgerundet (auf 8er-Schritte). Produkte ohne Bestellhistorie in den letzten 18 Monaten erscheinen nicht im Hauptangebot. In der Produktzeile wird die Kauffrequenz angezeigt: „Letzte 18 Monate: X× gekauft".

Erforderliche Datenpunkte pro Kunde:

  • order_history_549 — Produkt-IDs und Kauffrequenz der letzten 549 Tage (18 Monate)
  • suggested_quantity — Vorschlagsmenge (kann initial = Kauffrequenz × Aufrundungsfaktor sein, z. B. auf 8er-Schritte)

Bestellübergabe und Buchhaltung

Gratisprodukte werden nicht als separate Warenkorbpositionen übergeben. Der Kunde zahlt die volle Menge der bezahlten Stück, das Lager versendet die Gesamtmenge (bezahlt + gratis).

chargedQty = quantity                // Berechnungsgrundlage: nur bezahlte Menge
shippedQty = quantity + freeItems    // Tatsächlich versendete Menge
discount   = freeItems × unitPrice   // Naturalrabatt-Betrag (für Buchhaltung)

Empfohlene Repräsentation in Shopify-Order:

  • Order line item: quantity (bezahlt), Standard-Preis pro Einheit, ergänzendes Property free_items: N
  • Fulfillment: Pickliste wird aus quantity + free_items generiert
  • Buchhaltung: separater Buchungsposten natural_rabatt pro Order-Item für korrekte Margen-Ausweisung

Zusammenfassung der Implementierungs-Schritte

  1. Konstanten: Beide Tabellen (A + B) als feste Konfiguration ins Backend einbringen — nicht als Theme-Variable, sondern als App-/Function-Konstante.
  2. Product Metafield inuvet.pricing_model anlegen, Default A.
  3. Customer Tag group-customer in der CRM-Pflege etablieren.
  4. Berechnungsfunktion server­seitig implementieren — `freeItemsA`, `freeItemsB`, `getHint` analog zu den oben gezeigten JS-Skizzen.
  5. Order-Erstellung: free_items als Line-Item-Property setzen, Fulfillment-Hook für die Pickliste anpassen.
  6. Frontend: bezieht alle berechneten Werte über eine API-Antwort — keine Geschäftslogik im Theme. Hint-Text und Gratis-Badge werden serverseitig befüllt geliefert.
  7. Anzeige-Bedingung serverseitig prüfen: resolveBundle() auf 18-Monate-Historie — kein Rabatt im Pool → Bundle-Section nicht ausgeben. Die Entscheidung ist einmalig beim Load; das Frontend ändert die Sichtbarkeit danach nicht mehr.
